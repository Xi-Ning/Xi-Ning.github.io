# 1.1 考虑用静态工厂方法代替构造器

# 1.1.1 静态工厂方法

静态工厂方法（static factory method），只是一个返回类实例的静态方法。如：

public static Boolean valueOf(boolean b) {
    return b ? Boolean.TRUE : Boolean.FALSE;
}

# 1.1.2 与构造器相比，静态工厂方法的优势

优势1：有名称

  • 一个类只能提供一个带有指定签名的构造器，提供多个不同的构造器时，则需要改变参数列表；静态工厂方法则可以提供多个参数列表相同的方法，并使用方法名来突出它们之间的区别。

  • 类在提供多个构造器的情况下，若没有API文档，往往会不知所云，容易错用；静态工厂方法则可以通过方法名称使得方法含义更为清晰。

优势2：不必在每次调用时都创建一个新对象

  • 不可变类可以使用预先构件好的实例，或者将构建好的实例缓存起来，进行重复利用，从而避免创建不必要的重复对象。

  • 为重复的调用返回相同的对象，即能够做到实例受控（单例或不可实例化）。

优势3：可以返回原返回类型的任何子类型的对象，有更大的灵活性

  • 基于接口的框架（如：Collections）

    • 接口作为静态工厂方法的返回类型

    • 接口内部不能有静态方法，因此有如下惯例，接口Type的静态工厂方法被放在一个名为Types的不可实例化的类中

  • 服务提供者框架（如：JDBC）

    • 基础：静态工厂方法返回的对象所属的类，在编写包含该静态工厂方法的类时可以不必存在

    • 概念：多个服务提供者实现一个服务，系统为客户端提供多个实现，并把他们从多个实现中解耦出来

    • 构成组件：

      • 服务接口

      • 提供者注册API

      • 服务访问API

      • 服务提供者接口（可选）

// 服务接口
public interface Service {
}

// 服务提供者接口
public interface Provider {
    Service newService();
}

// 包含提供者注册API、服务访问API的不可实例化的类
public class Services {
    private Services() {}
    
    private static final Map<String, Provider> providers = new ConcurrentHashMap<String, Provider>();
    public static final String DEFAULT_PROVIDER_NAME = "def";
    
    // 提供者注册API
    public static void registerDefaultProvider(Provider p) {
        registerProvider(DEFAULT_PROVIDER_NAME, p);
    }
    public static void registerProvider(String name, Provider p) {
        providers.put(name, p);
    }
    
    // 服务访问API
    public static Service newInstance() {
        return newInstance(DEFAULT_PROVIDER_NAME);
    }
    public static Service newInstance(String name) {
        Provider p = providers.get(name);
        if(p == null) {
            throw new IllegalArgumentException("No provider registered with name:" + name);
        }
        return p.newService();
    }
}

# 1.1.3 静态工厂方法的缺点

缺点1：类如果不含有公有的或受保护的构造器，就不能被子类化

对于静态工厂方法所返回的非公有类，同样不能被子类化。（但恰巧：我们鼓励使用复合而非继承）

缺点2：静态工厂方法与其他的静态方法实际上没有区别

在API文档中，静态工厂方法没有像构造器那样明确标识出来。可以通过标准的命名习惯来弥补这一劣势：

  • valueOf：实际上是类型转换方法

  • of：valueOf的一种简洁替代

  • getInstance

  • newInstance：确保每个返回的实例都与其他实例不同

  • getType：Type表示返回的对象类型，用于工厂方法处于不同的类中

  • newType

# 1.2 遇到多个构造器参数时要考虑用构建器

# 1.2.1 静态工厂方法和构造器共同的局限性

静态工厂方法和构造器都不能很好的扩展大量的可选参数。

# 1.2.2 重叠构造器模式

重叠构造器模式：第一个构造器只有必要参数，第二个构造器有一个可选参数，第三个构造器有两个可选参数，以此类推，最后一个构造器包含所有构造参数。

public class Person {
    private String name;
    private String sex;
    private Date birthday;
    private String telephone;
    
    public Person(String name) {
        this(name, null);
    }
    public Person(String name, String sex) {
        this(name, sex, null);
    }
    public Person(String name, String sex, Date birthday) {
        this(name, sex, birthday, null);
    }
    public Person(String name, String sex, Date birthday, String telephone) {
        this.name = name;
        this.sex = sex;
        this.birthday = birthday;
        this.telephone = telephone;
    }
}
